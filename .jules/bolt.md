## 2025-01-28 - Optimizing `getSolicitacao` and `getSeparadores` with Prisma `where` Clauses
**Learning:** In Prisma setups, it's a common anti-pattern to fetch entire tables (e.g., `await prisma.user.findMany()`) into memory purely to check lengths or to apply JavaScript `.filter()` functions on large datasets (e.g., filtering `aprovado: false`). This over-fetches data, wastes database bandwidth, and causes severe memory spikes on the application level.
**Action:** Always verify if JavaScript-level filtering (`.filter()`) or unconstrained `findMany()` calls can be pushed down to the database level using Prisma's `where` clauses (e.g., `where: { role: 'SEPARADOR' }` or `where: { aprovado: false }`). This is particularly crucial for performance when returning large result sets from `PrismaService` methods.
