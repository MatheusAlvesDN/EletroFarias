## 2025-03-17 - Fix hardcoded JWT secret fallback in AuthModule
**Vulnerability:** The application was falling back to an insecure, hardcoded default `'dev-secret'` if the `JWT_SECRET` environment variable was not present. This was done in both `JwtModule.register()` and `JwtStrategy`.
**Learning:** In NestJS `AuthModule`, configuring `JwtModule` securely requires using `JwtModule.registerAsync` with `ConfigModule` and `ConfigService` to correctly and safely retrieve environment variables (e.g., `JWT_SECRET`). Similarly, `PassportStrategy` subclasses (e.g., `JwtStrategy`) require `ConfigService` injected in the constructor (without access modifiers like `private`) to securely fetch secrets before calling `super()`. Silent fallbacks to insecure keys expose token generation and verification to attacks.
**Prevention:** Avoid defining default values for cryptographic keys in code. Always validate that critical security configurations are present during initialization using an asynchronous factory or constructor logic, and throw explicit errors if they are missing to enforce a fail-secure state.
