## 2024-05-24 - Fix Insecure Default JWT Secret
**Vulnerability:** The application was falling back to an insecure, hardcoded default secret (`'dev-secret'`) during JWT authentication initialization if `process.env.JWT_SECRET` was undefined. Additionally, the JWT module was directly registering the secret from `process.env` which could silently fall back to undefined if not verified carefully.
**Learning:** Hardcoding a fallback secret introduces a critical security risk where attackers who know or guess the default secret can forge valid JWTs and bypass authentication. NestJS dependency injection can fail to start up cleanly or lead to TypeScript errors when injecting dependencies without properly bypassing access modifiers in child classes inheriting from generic abstractions (like `PassportStrategy(Strategy)`), often leading developers to just rely on `process.env` and hardcoded fallbacks to bypass the hassle.
**Prevention:** Always fail securely by explicitly throwing an error at startup if critical secrets (like `JWT_SECRET`) are missing. Do not provide hardcoded default secrets for anything other than local testing (and even then, ensure it never leaks to production through logical fallbacks). Use `JwtModule.registerAsync` to safely load secrets with `ConfigService` and properly validate their existence before starting the module. When extending classes like `PassportStrategy`, remember that you can inject configuration dependencies (e.g., `constructor(configService: ConfigService)`) without an access modifier to use them synchronously before calling `super()`.
