## 2025-02-21 - [Prevent Top-Level Env Checks in NestJS Modules]
**Vulnerability:** A hardcoded default secret (`'dev-secret'`) was used as a fallback if `JWT_SECRET` wasn't set, leading to severe token forgery risks.
**Learning:** During remediation, performing a top-level `if (!process.env.JWT_SECRET) throw new Error(...)` inside `auth.module.ts` crashed the application on startup. NestJS relies on its `ConfigModule` to populate the environment dynamically, making static import-time checks fail in normal usage. Additionally, inside `PassportStrategy` constructors with parameter properties (`private config: ConfigService`), one cannot call `config.get()` before `super()`.
**Prevention:** Always use `JwtModule.registerAsync` and inject `ConfigService` to perform deferred secret validation inside the `useFactory` block. When extending `PassportStrategy`, pass `configService` without an access modifier to the constructor to allow reading values before the mandatory `super()` call.
